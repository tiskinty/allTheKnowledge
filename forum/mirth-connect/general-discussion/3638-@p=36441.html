    Which version control option is correct with Mirth (as it appears all configurations are stored within the Mirth database:

    1. If the files exist on the file system natively which (directory) would I point to so I could version Mirth code within our Subversion system?
    2. Is the only means of versioning things exporting the structures to a directory and doing a manual check-in?

    The sane way to do it is option #2. Export your channels someplace, and manage them in your source control system from there. It takes some discipline to make sure you are updating and importing religiously, but it works.

    #1 could be done if you were using Derby, but I wouldn't recommend it. You'd basically have a large binary blob (the Derby files) in SVN. It's possible, but I can't think of a good reason to do it and lots of reasons not to.

    There is an open ticket in JIRA to add native SVN support to Mirth.

              I'm wondering if a viable option would be an automated export rather than full on source control integration.

              The process would look like this:
              1. Admin sets up a channel
              2. Admin selects "automatic export"
              3. Admin is prompted to enter a file path to export to
              4. Admin is prompted to enter an optional command to run after the export. This command runs in the path where the automatic export occurred.
              5. Whenever a channel is saved it is written to that path
              6. Whenever a channel is opened for editing, if the channel is older than the file the admin is prompted to import and overwrite

              This offers several advantages:
              1. No new significant code. Mirth already imports/exports files, checks timestamps, etc. Our wonderful developers do not need to learn some SVN library.
              2. No worries about what source control system to use. I like git. SVN is popular. CVS for the old schoolers. VSS for the MS shops. Many other systems. Too many to learn and support. Someone will always be unhappy that we don't support their system, this avoids that.
              3. Commits can still be done automatically using the command.

              Disadvantages:
              1. Making system commands is not ideal
              2. Source control admins should ensure that a given instance of Mirth should only have access to a few channels with commit rights.
              3. Still requires some thought on the part of the admin to check diffs before allowing an import

              So its not perfect. But it might work.
