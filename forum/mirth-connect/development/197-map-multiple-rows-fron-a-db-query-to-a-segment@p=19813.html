              Re: Map Multiple Rows fron a DB Query to a Segment

              Ok I see two things that you can do.

              Your hl7_lines table has many rows. Each row has one piece of hl7 information. So in your example, how do you know when you have all of the information for Pedro Perez and start reading information for Luis Sanchez?? It would be better to have an hl7_patient_info table with columns for Family Name, Given Name, DOB, Address...... then you select one row and map each row.

              The other option will work if you are guaranteed to have information on one person and only one person in that table, if you don&#039;t this will get wierd. I think this is the right query, I know it is close but might need some tuning. I do not advise using this, if at all possible redesign the DB.

              Code:
              SELECT family.value, given.value, dob.value, .....
FROM hl7_lines family
JOIN hl7_lines given ON family.id + 1 = given.id
JOIN hl7_lines dob ON given.id + 1 = dob
WHERE
family.name = &#039;Family Name&#039;
AND
given.name = &#039;Given Name&#039;
AND
dob.name = &#039;Date of Birth&#039;
              This is not a good way to solve the problem, it relies on having information on one person in the table or that information on people is always entered in order. That probably will not happen in a real-world database. If you are stuck with tables like that it will work, but it will also run very slowly and probably break often.

              I still recommend that you think about your database design. Mirth will do a lot of work for you if you already have all of this information in a database. You would need a table that Mirth can watch to know when to send a message. For example,
              Code:
              hl7_messages
<message type> <id to send> <sent>
              When you want to send an ADTA08 message on patient #1236 you would fill that table with ADTA08, 1236, 0. Then Mirth would read that, get information about patient #1236 from your regular database, map it into an ADTA08 message, then set sent = 1 so you know Mirth sent the message.

              I hope this helps. I don&#039;t know if you are stuck with the original tables or if you are working on a new design. Mirth is such a good tool because it does a lot of the work for you and you don&#039;t need to add a lot of new tables to an existing database to make it work.

                  I think your problem has an easy solution in mirth:

                  1) Create one Mirth channel for every event (or group of events) you want to receive, with the next query sentence:
                  Code:
                  select codigo,tipo,evento,fecha,hora,origen,destino,notificado, from hl7_pendiente where evento = &#039;A01&#039; and notificado <>?N?

                  UPDATE hl7_pendiente set notificado=?S? where codigo=${codigo}
