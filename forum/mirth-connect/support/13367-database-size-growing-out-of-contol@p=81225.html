    Database size growing out of contol

    06-16-2015, 06:36 AM
    I recently upgraded a Mirth Instance from 2.2.3 to 3.2.1 about a week ago.

    I'm using SQL Server 2008 R2 as the backend. Lots of changes between these two versions, and maybe I've missed something.

    But what I'm having the most issues with is the size of the SQL Server database. It is growing way too fast.

    Things I've tried:

    (1) Setting the Message Storage level down from Development to Production on high volume channels
    (2) Setting the Data Pruner to Prune Events at 3 days, setting the block size to 1000. It currently runs daily
    (3) Shrinking the SQL Server Database

    I think I've narrowed it down to the tables that contain the messages. In the old Mirth version, there used to be a single table called Messages, but now it seems the structure has changed and there are multiple tables (D_Mx, D_MAx, D_MCx, D_MCMx, D_MMx, D_MSx, D_MSQx) each having about 43 "x versions".

    One particular one seems to be causing the issue (I'm guessing this is one of my channels that has high volume; roughly 40000 messages/day)

    The D_MC21 table has over 6 million rows and is growing fast. Way too fast. The Mirth database is well over 100 GB in under a week.

    My Channel Message pruner is set at Prune Metadata older than 5 days.

    Are there any other pruning settings I can set? Surely this isn't normal behavior.

    Admittedly, I'm using an old method in one of my channels and receive this message in the Server Log for every message I receive:

    ===========================
    [2015-06-16 10:31:49,252] ERROR (com.mirth.connect.server.userutil.SerializerFacto ry:247): The getHL7Serializer() method is deprecated and will soon be removed. Please use the "Convert HL7 v2.x" templates from the References tab instead. Look at the tooltips to see the available property keys. The new method will strip namespaces by default unless the 'stripNamespaces' property is set to false.
    ===========================

    I just haven't found the time to fix it, because I'm trying to address the growth issue. I don't know if this is causing the issue or not.

    Looking for any guidance possible....

    Thank you.
    Walter

    In 3.x message data is normalized into several tables for metadata, content, attachments, etc. It's also typical that more data gets stored on average, simply because we added additional content types, dates, and other things to messages.

    if the D_MC21 table in particular is a problem, I'd recommend looking at the channel itself. On the Channels view you can right-click on the table header (or click the table controls in the top-right) and enable the Local ID, which will match with the number attached to the database tables. Post that channel if you can, making sure to scrub any PHI/passwords in it. How many destinations does it have? Are you using filtering?

        What is the average size of the raw data going into that channel? Note that over the course of message processing, typically several copies of the data will be stored, for the raw, transformed, encoded, channel map etc. And that's for each and every connector. If your channel is receiving some large static block of data like an attached PDF or image, consider using attachments so that it will only be stored once, as opposed to many times.

        You can also turn the message storage down to Raw or lower, but then note that you won't be able to use destination queuing in that case. Another thing you can do is set that 5 day pruning to something more frequent like 1-2 days.

                  First off, I notice that you're putting a ton of variables into the channel map in the source transformer. Like around 170 different variables. That, in itself, isn't "bad", obviously you need them because you're inserting all that data into a database. But it does mean that a lot of data will be stored for the channel map, for each enabled connector.

                  Also, you are doing filtering on a few of your destinations. I'd recommend using the "destination set filtering" feature we added in 3.1: MIRTH-3387. There are templates in the Reference list or auto-completion dialog to help you out with that.

                  For example on the "Log EPIC Demographics to table" destination you're only accepting messages that have the A31 or A28 trigger, and have a patient class of I or V. Even though you're filtering that destination, data for it will still be stored in the database. Instead if you remove that destination from the DestinationSet object in the source transformer, then the message won't even flow through that destination in the first place, and data won't be stored for it.

                  Another further optimization you can make is to limit which variables you put in the channel map in the source transformer. Like I said you're putting 170 or so different variables in there. But in most of your enabled destinations, you're not even using most of them anyway. For example the "FileWriter log" destination doesn't use any of them. It's only the "Log EPIC Demographics to table" that uses the majority of the variables. In the source transformer, consider only placing variables in the channel map that you will actually be using in more than one destination (like "MRN"). Then on the destination side, if a variable is only going to be used in that destination and no others, then map it in that destination's transformer, and put it in the connector map (not the channel map. For example the "MSPQ2" variable is only used in one destination, no others.

                  Another questionable decision is building up a "CleanedHL7" variable and putting that into the channel map. You're essentially putting another entire copy of the raw message, and duplicating it again by storing it in the channel map.

                  All of this is to say that there's nothing inherently "wrong" with your channel per se. But the implementation decisions that were made affect how much data is stored. Storage is cheap, and generally you shouldn't have to worry about that aspect nowadays. But it sounds like your environment has a limited amount of space, so you may have to re-think the architecture so you aren't double-storing data as much as possible.

                      I had a similar problem with a channel that processed large delimited files.

                      I neglected to turn off message archiving on this channel, meaning the pruner wouldn't remove messages that errored.

                      There were quite a few over the months, but this was a utility channel (as in I was trying something), so I didn't get a chance to go back and look.

                      Suddenly the appliance was running out of disk space.

                      I took a look at the Mirth database and saw that one table (for the aforementioned channel) was taking up a great deal of disk space!

                      I went through my channels and turned off message archiving where appropriate, and was relieved to see plenty of disk space after the pruner ran.
