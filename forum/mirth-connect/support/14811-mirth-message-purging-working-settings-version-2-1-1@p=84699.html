    Mirth Message Purging Working? Settings? Version 2.1.1

    03-08-2016, 07:32 AM
    Our interface engine sends us a ton of message, and our Mirth configuration is set to purge those processed messages after 3 days. The problem we have is that because we receive such a high volume of traffic, the automatic purge process appears to stop running even before we get to that 3 day window.

    We see that the mirth MESSAGE table is filling up with almost 500,000 new records each day, and there are currently 12 million records in this table.

    We had this problem previously and were forced to save off the data, truncate the table, then start fresh with the hope that the purge would keep up and actually prune messages out before we reached an unmanageable state, but that hasn’t happened. The purge process is set to run hourly and remove records in blocks of 1000. It appears that it ran the first two days since the table truncate occurred and has failed ever since.

    We need to get the purge process running and ensure that it removes records we no longer need before we hit a point where queries and inserts fail due to timeouts. We had reached this point previously when we hit 50 million records in the MESSAGE table, and are quickly approaching it again.

    Have we hit some sort of limitation of the purge process here considering the volume of messages we are processing?

    We are running Mirth version 2.1.1 on a legacy Windows 2003 server, with 1 GB of memory and are giving the JVM the maximum amount of memory on this server.

    Any insight on how to fix this problem would be appreciated.
    Tags: activity log, message pruner, mirth 2.1.1, pruning block size, settings

                Mirth Message Purging Working Settings Version 2.1.1

                Hi,

                In order to get rid of this issue,

                You need to increase the number of day’s (from 3 days to appropriate number) settings of Prune metadata for each channel such that data pruner has less number of messages to prune. Secondly, make sure block size is in small chunk under prune settings. You need to monitor prune job to make sure it runs without any error. Once data pruner runs successfully and start pruning the messages, you can decrease the number of days setting gradually.

                Hope this helps.

                Thanks,
                Sachin.
