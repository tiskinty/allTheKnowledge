    I'v found out that the problem is within the BEP / OPB segments. Those segments start with : instead of + because those are iterating segments.

    Normal segment: i.e. PID looks like: PID+::+:::+'

    BEP:1:1:1+0+Fundus foto diab. zorggr.-FUND+TEKST++++++FUFOFA'
    BEP:1:1:2+0+Natrium-NA+135++++++NA B'
    BEP:1:1:3+0+Kalium-K+4.5++++++K B'

    Does anyone has any idea if Mirth can handle this in any way? It look likes this is a standard which is only commonly use here in the Netherlands.

          You could replace those matches in the preprocessor with a +, but why would you need to? If the segment was:

          ABC:1:1:1+Something'

          And you took out the :1:1:1, it would become:

          ABC+Something'

          If you instead replaced :1:1:1 with +, it would be:

          ABC++Something'

          Similarly, a segment like

          ABC:1:1:1'

          Would become:

          ABC+'

          Is that what you want? If so, that's a pretty easy change..

                Say you have some data that you want to map that comes in "ARA.1". So if your inbound message is:
                Code:
                ARA+value

                Code:
                ARA:1+value
                Now, if you change that to:
                Code:
                ARA+1+value
                then ARA.1 is now "1" instead of "value". And that will be true for every component in the segment...

                          Do you actually want to SEND the data after it's been "reversed" back to the original format?

                          If so, you'll need to do that in the transformer, rather than in the postprocessor. And so this begs the question: why not just use delimited text rather than EDI? That way you won't have to convert the message into EDI and then back.

                          Also, the problem with reversing the formats is that by converting the inbound data into EDI, you've actually lost some information. Not all segments have the ABC:1:1:1 format; some are like DEF+1+somethingelse, right? So once you've converted to ABC+1:1:1, you won't necessarily know which ones to convert back, and which ones to leave alone. One way to remedy this is, as I said, storing the raw message (before any altering at all) in the channel map in the preprocessor script, and then use that to "reverse" the final message at the end in your transformer. But, at this point, you're doing way too much work for something that can be solved so much more simply.

                          I reiterate my previous question: Do you really want to convert a generic segment ABC:1:1:1+something to ABC+1:1:1+something? Yes, it'll make it so that Mirth will accept it as a "valid" EDI message, but now "something" is located in the third field rather than the second, and everything else in the entire segment (and every other segment for which this happens) is skewed. It would seem much easier to just read in messages as delimited text. Or, if the client to which you're sending the final messages doesn't care about the ":1:1:1" that sometimes gets appended to the segment name, then you can just use the original replace statement, get rid of that stuff in the postprocessor, and treat the message as valid EDI with no problems.
