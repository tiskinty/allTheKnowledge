                    Very strange.

                    The error suggests a parsing error, this is the kind of errors you usually get when defined and passed data types missmatch or encoding problems. But in your channel Data Types seem to be set appropriately and I don't see anything strange in your message.

                    What if you try with another HL7 message, e.g. ADT or ORU or anything ? Something as simple as

                    MSH|^~\&|APP1|FAC1|APP2|FAC2|202102020000||ADT^A08 |12345678901|P|2.4|
                    EVN|A08|202102020000|||||
                    PID||111111|||name^given||19910101|M|||||||||||||| |||||||||||||||

                    What if you set *default* encoding instead of UTF-8, so that you'll use your computer Locales. Quite sure it won't change anything, but still maybe ...

                    And what if you try again from scratch ?

                    Edit : I didn't pay attention to this : your file is UTF-8 with BOM encoded, try to remove it in your preprocessor script (not sure if currently removed) :
                    Code:
                    return message.replace(/\uFEFF/g,'');
                    Edit2 : I've read your initial post once again very carefully : are you 100% sure that posted channel, message sample and error trace are accurate? Because actually you should not get that exception in your scenario.

                    Code:
                    99: msg = new XML(connectorMessage.getTransformedData());
                    This suggests that incoming data type is set to XML, so that a xml string comes in and is parsed into an XML / E4X object. But in your case you set HL7 as inbound data type. So a parsing error should be more type of "Error converting ER7 to XML".
