    You actually don't need to do any of that.

    On the destination sender, the IP and port accept velocity variables, so you can dynamically set them per message and let the single destination send to variable endpoint.

    For example in a transform set these based on where the message should go.

    $c('send_to_port', '7000') // obviously set this correctly
    $c('send_to_ip', '127.0.0.1') // obviously set this correctly

    And on the desination:

    Remote Address: ${send_to_ip}
    Remote Port: ${send_to_port}

          02-22-2021, 12:09 PM
          $c is channelMap
          $gc is globalChannelMap
          $g is globalMap

          I suggest $c because it is per message.

          So based on some field in the HL7 message you can set 'send_to_port' and 'send_to_ip' for that message.

          When the next message arrives you would set it again based on the settings for that message.

          Reminder

          $c('send_to_port', '7000') // obviously set this correctly
          $c('send_to_ip', '127.0.0.1') // obviously set this correctly

          Means the same exact thing as:

          channelMap.put('send_to_port', '7000')
          channelMap.put('send_to_ip', '127.0.0.1')

                  Second, it does not look to me like the substitution is happening in the remote address and remote port fields of the tcp (mllp) destination channel since I still get the "Remote port is invalid" error

                  Third, I am able to confirm the port is a string (see debugging below) and I've also tried casting it to a number.

                  Fourth, I've demonstrated some debugging tips below to answer my question about debugging javascript. If anyone has a better workflow please let me know.

                  Fifth, It turns out using a Map lead me astray but a POJsO (plain old javascript object) works in its place. I'm including the updated code below and the variable values.

                  Debugging:

                  You can view the channel variables in the Mappings section of your connector when viewing messages on your channel.

                  I changed the global script on my channel to try a pojso:

                  Code:
                  //// This script executes once when the channel is deployed
//// You only have access to the globalMap and globalChannelMap here to persist data
const portMap = new Map();
portMap.set('FakeA', ['127.0.0.1', '8087'])
portMap.set('FakeB', ['127.0.0.1', '8088'])
$gc('portMap', portMap)
const portRoutes = {'FakeA': ['127.0.0.1', '8087'],
                    'FakeB': ['127.0.0.1', '8088']}
$gc('portRoutes', portRoutes)
return;
                  I updated the javascript transformer step to set channel variables beginning with "debug_"

                  Code:
                  const portMap = $gc('portMap')
const portRoutes = $gc('portRoutes')

const msh_4_1 = validate(msg['sending_facility'], '', new Array())

$c('debug_msh_4_1', msh_4_1)

// Debug portMap
function mapToObj(map){
  const obj = {}
  for (let [k,v] of map)
    obj[k] = v
  return obj
}

var myJson = {};
myJson.myMap = mapToObj(portMap);
const jsonStrPortMap = JSON.stringify(myJson);

$c('debug_json_of_portMap', jsonStrPortMap)
var portMapHasIt = portMap.has(msh_4_1)
$c('debug_portMapHasIt', portMapHasIt)

// Debug portRoutes
$c('debug_portRoutes', portRoutes)
var pojsoHasIt = portRoutes.hasOwnProperty(msh_4_1)
$c('debug_pojsoHasIt', pojsoHasIt)

// Debug booleans
const myBoolTriple = msh_4_1 === "FakeB"
$c('debug_myBoolTriple', myBoolTriple)
const myBoolDouble = msh_4_1 == "FakeB"
$c('debug_myBoolDouble', myBoolDouble)


// if (portMap.has(msh_4_1)) {
if (portRoutes.hasOwnProperty(msh_4_1)) {
  const [port, ip] = portRoutes[msh_4_1]
  $c('send_to_port', port)
  $c('debug_type_port', typeof(port))
  $c('send_to_ip', ip)
  $c('debug_type_ip', typeof(ip))
} else {
  $c('send_to_port', 'NOT FOUND')
  $c('send_to_ip', 'NOT FOUND')
}
tmp['MSH']['MSH.4']['MSH.4.1'] = msh_4_1;
